O que é o OpenClaw?

Agente, não chatbot

ChatGPT responde. OpenClaw age. Executa tarefas realmente: controla browser, escreve ficheiros, corre comandos, envia mensagens.

Local-first & privado

Corre na tua máquina. Os dados nunca saem da tua rede. Trazes a tua própria API key (Claude, GPT, Ollama local...).

Via apps que já usas

Acedes por WhatsApp, Telegram, Slack, Discord, iMessage — onde quiseres. Um sistema, múltiplos canais.

História: Clawdbot → Moltbot → OpenClaw

Criado por Peter Steinberger em finais de 2025. 60K+ estrelas no GitHub em 72h. Em Fevereiro 2026 o criador juntou-se à OpenAI e o projeto passou para uma fundação open-source.

Tools & Skills

Distinção importante

Tools = capacidades nativas do sistema (o que o agente pode fazer). Skills = plugins/instruções que ensinam o agente a usar as tools de forma especializada. Skills são manuais de instruções; o acesso real é controlado por tools.allow.

Otimizar Custos da API

Baseado na análise do vídeo «I Cut My OpenClaw Costs by 95%» de Nick Puru e guias adicionais da comunidade.

O problema que a maioria não sabe que tem

O OpenClaw faz chamadas à API em background 24/7 mesmo sem estares a usar. Nick Puru descobriu que 85% do seu gasto de $50/semana era puro desperdício: 60% overhead invisível, 25% modelo errado para a tarefa, apenas 15% trabalho produtivo real.

Breakdown do Gasto Antes dos Fixes

Overhead puro (heartbeats + context loading) 60%
Spend mal direcionado (modelo premium para tarefas simples) 25%
Trabalho produtivo real 15%
Camada 1 Eliminar o Overhead Invisível Resolve os 60%
A

Redirecionar Heartbeat para Modelo Local (Ollama)

O heartbeat dispara a cada 30–60 min carregando o contexto COMPLETO (dezenas de milhar de tokens), 24/7. Fix: instalar Ollama + Llama 3B (2GB, gratuito) e apontar o heartbeat ao modelo local. Resultado: ~40% do spend vai a zero overnight.

B

Controlar o que Carrega no Contexto

Adicionar ao system prompt: no início de sessão carrega APENAS (1) ficheiro core, (2) perfil do utilizador, (3) identidade do agente, (4) notas de hoje. Todo o resto carrega sob demanda. Resultado: 40–60KB por interação → 6–10KB.

Camada 2 Gastar Melhor no Trabalho Real Resolve os 25% + baixa o resto 70–75%
C

Modelo Certo para a Tarefa Certa

Haiku como modelo padrão (10–12x mais barato que Sonnet, mais rápido). Escalar para Sonnet/Opus APENAS para: arquitetura, código de produção, segurança, raciocínio multi-step. Resultado: −70 a 75% no custo de trabalho produtivo.

D

Cortar os Ficheiros de Contexto

Nos ficheiros core: só o que o agente precisa AGORA. Contexto detalhado, histórico e notas → ficheiros separados que carregam sob demanda. Instruir o agente a reportar uso de tokens após cada tarefa.

E

Budget Guardrails

Adicionar ao system prompt: delay mínimo entre chamadas, máximo de web searches por batch, limites diários/mensais com alertas. Não ativar faturação automática até o sistema estar estável — carregar $5–10 manualmente e observar como drena.

Camada 3 O Multiplicador — Prompt Caching 90% de desconto no conteúdo estável
F

Prompt Caching

Primeira vez que envias contexto → pagas preço cheio. Chamadas seguintes dentro de 5 minutos → 90% de desconto no conteúdo em cache. Cache apenas conteúdo estável: instruções, perfil do utilizador, docs de referência. Não editar instruções a meio de sessão (quebra o cache). Amplifica todas as outras otimizações.

Casos Práticos por Setor

Caso 1 — Agência de Marketing Digital

Uma agência com 3 gestores de conta e 12 clientes ativos. Cada gestor passava 2h/dia a compilar relatórios manualmente a partir do Google Analytics, Meta Ads e LinkedIn.

Antes do OpenClaw

  • 2h/dia por gestor a copiar métricas para Excel
  • Erros frequentes por cópia manual
  • Relatórios enviados sempre com 1-2 dias de atraso
  • Clientes a pedir updates por WhatsApp constantemente

Depois do OpenClaw

  • Relatório automático toda segunda-feira às 8h
  • Dados de todas as plataformas consolidados em PDF
  • Enviado automaticamente para o cliente via email
  • Gestores respondem ao WhatsApp do cliente em 2 min com contexto pronto
Instrução enviada pelo gestor ao agente (WhatsApp)
"Toda segunda-feira às 8h, recolhe os dados de Google Analytics, Meta Ads e LinkedIn Ads do cliente XYZ referentes à semana anterior. Compila num relatório com as métricas principais (impressões, cliques, CTR, conversões, custo/conversão) e envia por email para o cliente com o assunto 'Relatório Semanal — [data]'. Se alguma métrica descer mais de 20% face à semana anterior, inclui um aviso em destaque e sugere 2 ações corretivas."

Caso 2 — Consultora / Freelancer

Consultor independente com 8 clientes ativos. Passava 3-4h/semana em admin: responder emails repetitivos, agendar reuniões, enviar follow-ups e atualizar o CRM.

Email Inteligente

O agente lê a caixa de entrada às 9h e categoriza: urgente, follow-up necessário, informativo, spam. Rascunha respostas para os emails rotineiros. O consultor revê e aprova em 15 minutos.

Agendamento Automático

Cliente envia WhatsApp: "Quero marcar uma reunião". O agente verifica o calendário, propõe 3 slots disponíveis, confirma e cria o evento com link Zoom incluído.

Follow-ups Automáticos

Se uma proposta enviada não tiver resposta em 5 dias, o agente envia automaticamente um follow-up personalizado. Sem intervenção humana.

CRM Atualizado

Após cada reunião, o agente lê as notas enviadas pelo consultor e atualiza automaticamente o registo no CRM (HubSpot/Notion) com o estado, próximos passos e data de follow-up.

Resultado: De 3-4h/semana de admin → 30 minutos de revisão. O consultor recuperou ~14h/mês para trabalho faturável.

Caso 6 — Complexidade Média: Negociação Autónoma de Fornecedores

Um freelancer precisava de renovar o seu equipamento fotográfico. Em vez de comparar preços manualmente, delegou ao OpenClaw toda a fase de pesquisa, contacto e negociação com fornecedores.

Inspirado em caso real documentado: utilizador delegou compra de viatura e obteve €4.200 abaixo do preço de tabela via negociação autónoma entre dealers.

Fluxo de Execução

Fase 1 — Pesquisa de Mercado
Agente pesquisa web por fornecedores que vendem o equipamento definido. Compila lista com preços, condições e contactos. Envia resumo ao utilizador para aprovação.
Fase 2 — Contacto Inicial
Para cada fornecedor aprovado, envia email de pedido de cotação com especificações precisas. Regista respostas e prazo máximo de 48h.
Fase 3 — Negociação Cruzada
Com cotações em mãos, reencaminha cada proposta para os concorrentes: "Recebi proposta de €X de fornecedor Y. Consegue melhorar?" Regista todas as contra-propostas.
Fase 4 — Decisão
Apresenta relatório final ao utilizador: tabela comparativa com preço, condições, prazo e reputação. Utilizador decide e agente formaliza a encomenda.

O que torna este caso "complexidade média"

  • Múltiplas fases sequenciais com dependências
  • Comunicação com múltiplas partes externas (fornecedores)
  • Lógica condicional: se não responder em 48h → follow-up automático
  • Checkpoint humano entre fases (aprovação de lista, decisão final)
  • Execução assíncrona ao longo de vários dias
Resultado típico: Poupança de 8-15% no preço final. Sem horas gastas em telefonemas e emails repetitivos.

Caso 7 — Complexidade Alta: Equipa Multi-Agente de Conteúdo SEO

Uma startup de SaaS criou uma equipa de 3 agentes OpenClaw especializados que trabalham em conjunto para produzir, otimizar e publicar conteúdo SEO de forma totalmente autónoma — do trend research ao artigo publicado.

Baseado em casos documentados na comunidade — utilizadores reportam aumentos mensuráveis de tráfego orgânico com pipelines equivalentes.

Arquitetura da Equipa

Agente Investigador

Executa diariamente às 7h

  • Monitoriza trends (Reddit, HN, X)
  • Pesquisa keywords em ascensão
  • Analisa conteúdo dos concorrentes
  • Envia briefing ao Agente Editor

Agente Editor

Ativado pelo briefing do Investigador

  • Seleciona o melhor tópico do briefing
  • Redige artigo SEO completo (1500+ palavras)
  • Inclui meta title, description, H1-H3
  • Envia draft ao Agente Publisher

Agente Publisher

Aguarda aprovação humana

  • Envia draft ao humano via Telegram
  • Aguarda "ok" ou feedback
  • Aplica revisões se necessário
  • Publica no CMS via API

Fluxo Completo — Da Ideia à Publicação

07:00 Investigador analisa Reddit/HN/X. Identifica: "Artigo sobre 'OpenClaw vs n8n para não-técnicos' tem 0 resultados SEO mas 800 pesquisas/mês". Envia briefing.
07:15 Editor recebe briefing. Redige artigo de 1800 palavras com estrutura SEO, 3 H2, exemplos práticos, FAQ e CTA.
07:45 Publisher envia draft via Telegram: "Novo artigo pronto — 'OpenClaw vs n8n: Qual Escolher em 2026?'. Aprovar? [Sim] [Editar] [Rejeitar]"
08:10 Humano lê no Telegram, responde "Sim". Publisher publica no WordPress via API com slug, categoria e tags corretas.
08:11 Artigo publicado. Total de tempo humano investido: 2 minutos de revisão. Total de artigos publicados por semana com este setup: 5.

O que torna este caso "complexidade alta"

  • 3 agentes autónomos com papéis distintos e comunicação entre si
  • Lógica de ramificação: aprovação, edição ou rejeição geram fluxos diferentes
  • Integração com APIs externas: CMS, Reddit, ferramentas SEO, Telegram
  • Execução autónoma diária via cron sem intervenção humana (exceto aprovação)
  • Memória partilhada entre agentes: o que foi publicado, o que foi rejeitado
  • Checkpoint humano obrigatório antes de publicar — controlo de qualidade
Resultado reportado: 5 artigos SEO/semana publicados com 2 min de supervisão humana por artigo. Aumento de tráfego orgânico em 3-4 meses. Custo total do sistema: ~€15-30/mês em API.

Segurança & Boas Práticas

Riscos Principais

Configurações Incorretas

Gateway exposto à internet, allowlists demasiado permissivas, tools com acesso total desnecessário.

Prompt Injection

Inputs maliciosos disfarçados de comandos. Websites maliciosos podem tentar dar instruções ao agente.

Skills Maliciosas

341 skills maliciosas encontradas no ClawHub (ClawHavoc, Fev 2026). Verificar código fonte antes de instalar qualquer skill.

Boas Práticas

  • Contas dedicadas para o agente (não contas pessoais)
  • Nunca expor porta 18789 diretamente à internet
  • Usar Tailscale ou SSH tunnel para acesso remoto
  • Deploy em Docker com isolamento de rede (enterprise)
  • Rotar API keys regularmente
  • Perfil "minimal" ou "coding" — não "full" sem necessidade
  • Verificar repositório GitHub + código de cada skill
  • Revogar acesso de skills não utilizadas
  • Ativar Control UI dashboard para monitorização
  • Não ativar faturação automática no início

Checklist Enterprise (antes de ir a produção)

  • Docker containerização configurada
  • Contas de serviço dedicadas
  • Credenciais rotativas programadas
  • Audit logging ativo
  • Gateway não exposto publicamente
  • VPN ou Tailscale para acesso remoto
  • Allowlists de remetentes configuradas
  • Todas as skills revisadas e aprovadas
  • Plano de rollback documentado
  • Responsável de segurança designado
